เกี่ยวกับโปรเจกต์นี้

โปรเจกต์นี้เป็นแอปพลิเคชันสำหรับจัดการข้อมูลนักศึกษาพร้อมรูปภาพโปรไฟล์ พัฒนาด้วย Flutter และเชื่อมต่อกับฐานข้อมูล Cloud Firestore ของ Firebase โดยตรง จุดประสงค์หลักคือการฝึกทำระบบ CRUD ครบวงจร ได้แก่ การเพิ่ม การอ่าน การแก้ไข และการลบข้อมูล ผ่านการเชื่อมต่อกับฐานข้อมูลบนคลาวด์จริง และแสดงผลข้อมูลแบบเรียลไทม์

ตัวแอปสามารถเพิ่มข้อมูลนักศึกษาใหม่ผ่านฟอร์มที่มีการตรวจสอบความถูกต้องของข้อมูล แสดงรายชื่อนักศึกษาทั้งหมดพร้อมรูปโปรไฟล์แบบวงกลม แก้ไขข้อมูลเดิมได้โดยการแตะที่รายชื่อ และลบข้อมูลได้โดยมีหน้าต่างยืนยันก่อนลบทุกครั้ง เมื่อมีการเปลี่ยนแปลงข้อมูลบนฐานข้อมูล หน้าจอแอปจะอัปเดตตามทันทีโดยไม่ต้องรีเฟรช


โครงสร้างข้อมูล

ข้อมูลนักศึกษาถูกเก็บใน Collection ชื่อ students บน Firestore โดยแต่ละคนมีข้อมูล 5 อย่างคือ รหัสนักศึกษา (studentId) ชื่อนามสกุล (name) สาขาวิชา (major) เกรดเฉลี่ย (gpa) และลิงก์รูปโปรไฟล์ (imageUrl) ซึ่งรูปโปรไฟล์สร้างจากบริการรูปภาพจำลองโดยอ้างอิงจากรหัสนักศึกษาของแต่ละคน


โครงสร้างไฟล์ในโปรเจกต์

โค้ดทั้งหมดอยู่ในโฟลเดอร์ lib โดยแบ่งหน้าที่กันชัดเจน ไฟล์ main.dart เป็นจุดเริ่มต้นของแอปและทำหน้าที่เชื่อมต่อ Firebase ก่อนเริ่มทำงาน ไฟล์ในโฟลเดอร์ models เก็บคลาส Student ที่เป็นตัวแทนข้อมูลนักศึกษาพร้อมเมท็อดแปลงข้อมูลไปกลับระหว่างแอปกับ Firestore ไฟล์ในโฟลเดอร์ services เก็บ StudentService ที่รวมคำสั่งติดต่อกับฐานข้อมูลทั้งหมดไว้ที่เดียว และไฟล์ในโฟลเดอร์ pages เก็บหน้าจอสองหน้าคือหน้าแสดงรายชื่อและหน้าฟอร์มเพิ่มแก้ไขข้อมูล นอกจากนี้ยังมีไฟล์ทดสอบใน test สำหรับทดสอบการทำงานของคลาส Student


วิธีติดตั้งและรัน

ขั้นแรกติดตั้ง dependencies ด้วยคำสั่ง flutter pub get จากนั้นเชื่อมต่อ Firebase ด้วยคำสั่ง flutterfire configure เพื่อสร้างไฟล์ตั้งค่า แล้วเข้าไปเปิดใช้งาน Cloud Firestore ใน Firebase Console โดยเลือกโหมดทดสอบ สุดท้ายรันแอปด้วยคำสั่ง flutter run หากต้องการทดสอบการทำงานของโค้ดโดยไม่ต้องเชื่อมต่อฐานข้อมูล สามารถรัน flutter test ได้


คำตอบคำถามทบทวนท้ายบทเรียน

ข้อ 1 เหตุใดข้อมูลบนหน้าจอแอปจึงเปลี่ยนแปลงทันทีเมื่อมีการอัปเดตข้อมูลบน Firebase Console โดยไม่ต้องดึงข้อมูลใหม่ และกลไกของ StreamBuilder ร่วมกับ Firestore ทำงานอย่างไร

สาเหตุที่ข้อมูลเปลี่ยนทันทีเป็นเพราะแอปใช้ StreamBuilder ทำงานร่วมกับเมท็อด snapshots ของ Firestore ซึ่งเป็นการทำงานแบบที่ฐานข้อมูลส่งข้อมูลมาให้เอง ไม่ใช่แบบที่แอปต้องคอยไปดึงข้อมูลเอง

การทำงานคือ เมื่อเรียกใช้ snapshots มันจะไม่ได้ดึงข้อมูลเพียงครั้งเดียวแล้วจบ แต่จะเปิดการเชื่อมต่อค้างไว้กับเซิร์ฟเวอร์ตลอดเวลาเพื่อคอยฟังการเปลี่ยนแปลง เมื่อข้อมูลใน Collection students บนเซิร์ฟเวอร์มีการเปลี่ยนแปลง ไม่ว่าจะแก้ไขจาก Console หรือจากที่อื่น เซิร์ฟเวอร์จะส่งข้อมูลชุดใหม่กลับมาให้แอปทันที ทำให้ StreamBuilder รับรู้ว่ามีข้อมูลใหม่และสั่งวาดหน้าจอส่วนนั้นใหม่โดยอัตโนมัติ

เปรียบเทียบง่ายๆ ได้กับการสมัครรับข่าวสาร เมื่อมีข่าวใหม่ระบบจะส่งมาให้เราถึงที่ทันที เราจึงไม่ต้องคอยไปเปิดดูเองว่ามีข่าวใหม่หรือยัง ด้วยเหตุนี้จึงไม่จำเป็นต้องรีเฟรชหน้าจอ

ข้อ 2 ในกรณีที่อุปกรณ์ของผู้ใช้ขาดการเชื่อมต่ออินเทอร์เน็ต ฟีเจอร์ Local Persistence ของ Cloud Firestore จะจัดการกับข้อมูลที่เพิ่มหรือแก้ไขในขณะนั้นอย่างไร

Firestore บนมือถือมีการเปิดใช้ Local Persistence หรือการเก็บข้อมูลไว้ในเครื่องเป็นค่าเริ่มต้นอยู่แล้ว เมื่ออุปกรณ์ไม่มีอินเทอร์เน็ต ในส่วนของการอ่านข้อมูล แอปยังสามารถดึงข้อมูลจากที่เก็บไว้ในเครื่องมาแสดงได้ตามปกติ ไม่ค้างและใช้งานต่อได้

ส่วนการเพิ่มหรือแก้ไขหรือลบข้อมูลในขณะที่ไม่มีเน็ต ตัว Firestore จะเขียนข้อมูลลงในเครื่องก่อนทันที ทำให้หน้าจอแสดงผลเหมือนบันทึกสำเร็จแล้ว พร้อมกับเก็บคำสั่งที่ทำไว้เข้าคิวรอไว้ในเครื่อง เมื่ออุปกรณ์กลับมาเชื่อมต่ออินเทอร์เน็ตได้อีกครั้ง Firestore จะส่งคำสั่งที่ค้างอยู่ในคิวขึ้นไปยังเซิร์ฟเวอร์โดยอัตโนมัติตามลำดับ ทำให้ข้อมูลบนคลาวด์ตรงกับที่ผู้ใช้ทำไว้ตอนออฟไลน์ ผลก็คือผู้ใช้สามารถทำงานต่อได้ราบรื่นเหมือนออนไลน์อยู่ตลอด และข้อมูลจะไม่สูญหาย

ข้อ 3 หากต้องการอัปโหลดไฟล์ภาพจริงที่ถ่ายจากกล้องในเครื่องเข้าไปเก็บบนระบบออนไลน์ของ Firebase จะต้องเลือกใช้บริการส่วนใด

ต้องใช้บริการ Firebase Cloud Storage ซึ่งเป็นบริการที่ออกแบบมาสำหรับเก็บไฟล์ขนาดใหญ่โดยเฉพาะ เช่น รูปภาพ วิดีโอ หรือเอกสาร

วิธีการทำงานคือ อัปโหลดไฟล์ภาพจากกล้องขึ้นไปเก็บที่ Cloud Storage ก่อน จากนั้น Storage จะคืนลิงก์สำหรับดาวน์โหลดไฟล์กลับมา แล้วเราจึงนำลิงก์นั้นไปเก็บในฟิลด์ imageUrl ของข้อมูลใน Firestore อีกที

เหตุผลที่ต้องแยกใช้สองบริการนี้เป็นเพราะ Firestore เหมาะกับการเก็บข้อมูลที่เป็นข้อความหรือตัวเลขที่มีโครงสร้าง ไม่เหมาะกับการเก็บไฟล์ขนาดใหญ่ ในขณะที่ Cloud Storage ถูกออกแบบมาเพื่อเก็บไฟล์โดยเฉพาะ ทั้งสองบริการจึงทำงานเสริมกัน โดย Storage ทำหน้าที่เก็บไฟล์จริง ส่วน Firestore เก็บเพียงลิงก์ที่ชี้ไปยังไฟล์นั้น


หมายเหตุ

กฎความปลอดภัยของโปรเจกต์นี้ตั้งเป็นโหมดทดสอบซึ่งเปิดให้อ่านและเขียนข้อมูลได้อย่างอิสระ เหมาะสำหรับการเรียนและการทดลองเท่านั้น ไม่เหมาะกับการนำไปใช้งานจริง


เทคโนโลยีที่ใช้

พัฒนาด้วย Flutter ภาษา Dart ใช้ Cloud Firestore เป็นฐานข้อมูล และใช้แพ็กเกจ firebase_core กับ cloud_firestore ในการเชื่อมต่อกับ Firebase
